ИИ-Агент Оркестратор

Единый агент, который декомпозирует задачу на атомы, последовательно переключается между ролями Аналитик → Разработчик → Тестировщик, валидирует артефакты на каждом gate и ведёт git по конвенциям методологии.

Классический процесс разработки требует координации трёх ролей. В гибридной AI-разработке агент берёт на себя роль оркестратора — он не заменяет людей, а структурирует работу так, что каждый артефакт проходит через формальные gate-проверки прежде чем попасть к следующей роли.

Ключевой принцип: агент не пропускает роли. Даже для "простой" задачи пайплайн проходит полностью: анализ → реализация → тестирование. Gate между ролями — обязателен.

Каждая роль читает свой reference-файл перед началом работы. Reference содержит: чек-листы, матрицы решений, примеры заполнения каждой секции, частые ошибки и anti-patterns.

Workflow: от задачи к коду

Оркестратор

Шаг 0 — Декомпозиция

Определить домен, тип изменения, change_class. Если задача затрагивает несколько use-case — декомпозировать на отдельные атомы. Показать план пользователю.

Аналитик

Шаг 1 — Анализ требований

Прочитать references/analyst.md. Создать spec.md: Intent, Domain Rules, Acceptance Criteria (Gherkin), Domain Model Touch, Constraints. Самопроверка по Gate A.

Разработчик

Шаг 2 — Реализация

Прочитать references/developer.md. Изучить атом. Написать Tech Spec (API, схемы, архитектура), код с маркерами DR-N, Implementation Notes. Самопроверка по Gate B.

Тестировщик

Шаг 3 — Тестирование

Прочитать references/tester.md. Составить Coverage Matrix. Расширить AC edge-cases. Написать Test Plan с конкретными данными. Написать тестовый код. Самопроверка по Gate C.

Оркестратор

Шаг 4 — Финальная валидация

Все gates пройдены? Файлы в правильной структуре? Git-конвенции соблюдены? Open Questions закрыты? → Итоговый отчёт пользователю.

Роль: Аналитик

PM / Business Analyst

Аналитик превращает бизнес-потребность в формализованный атом. Его секции технологически агностичны — никаких фреймворков, баз данных, API-эндпоинтов.

Владеет секциями

§ Intent
Бизнес-намерение в 1–3 предложениях
§ Domain Rules
DR-N: проверяемые инварианты
§ Acceptance Criteria
Gherkin — ЧТО, не КАК
§ Domain Model Touch
Агрегаты, события, инварианты
§ Constraints
[PERF] [SEC] [IDMP] [A11Y]
§ Open Questions
OQ-N: blocking / non-blocking
Anti-pattern: DR без последствий. "Email должен быть уникален" — а что если нет? Каждое правило = условие + реакция системы при нарушении.

Роль: Разработчик

Developer

Разработчик получает атом после Gate A и превращает его в работающий код. Атом — контракт. DR и AC — это ТЗ. Если что-то неясно — не додумывай, открой Open Question.

Владеет секциями

§ Tech Spec
API-контракты, схемы, архитектура
§ Implementation Notes
Trade-offs, deps, limitations, file map
маркировка DR в коде
def register_user(email, password): # DR-1: Email must be unique if user_repo.exists_by_email(email): raise EmailAlreadyExistsError(email) # DR-2: Password policy violations = password_policy.validate(password) if violations: raise WeakPasswordError(violations) user = User.create(email, hash_password(password)) event_bus.publish(UserRegistered(user.id, email))

Роль: Тестировщик

QA Engineer

Тестировщик получает атом после Gate B. AC Аналитика — минимум, не максимум. Тестировщик расширяет покрытие: edge-cases, boundary conditions, concurrency, injection, encoding.

Владеет секциями

§ Test Plan
TC-NNN: конкретные данные, шаги, ожидания
§ AC Review
Расширяет AC Аналитика edge-cases

Категории edge-cases

обязательные проверки
1. Boundary values — мин/макс значения полей 2. Empty/null/missing — пустые строки, null, без поля 3. Format violations — невалидные форматы 4. Injection — SQL, XSS 5. Concurrency — одновременные запросы 6. Idempotency — повторная отправка 7. Rate limiting — граничные значения 8. Encoding — Unicode, emoji, RTL 9. Error recovery — БД down, bus down 10. Timeout — превышение PERF-лимита

Анатомия атома

Каждый атом = YAML frontmatter + секции Markdown. Секции идут от абстрактного к конкретному — принцип progressive disclosure.

frontmatter атома
--- id: AUTH-REG-001 type: use-case # system | domain | aggregate | use-case | scenario parent: AUTH title: "Регистрация пользователя" status: draft # draft → review → ready → implemented → deprecated change_class: additive # breaking | additive | compatible actors: [Guest] emits: [UserRegistered] consumes: [] owners: analyst: "@anya" developer: "@dev" tester: "@qa" ---
секции атома — кто владеет
## § Intent ← Аналитик: бизнес-намерение ## § Domain Rules ← Аналитик: DR-N инварианты ## § Acceptance Criteria ← Тестировщик + Аналитик: Gherkin ## § Domain Model Touch ← Аналитик + Dev: агрегаты, события ## § Constraints ← Аналитик: NFR [PERF] [SEC] [IDMP] [A11Y] ## § Open Questions ← Все: OQ-N blocking / non-blocking ## § Decision Log ← Все: DL-N решения с датой ## § Tech Spec ← Разработчик: API, схемы, архитектура ## § Test Plan ← Тестировщик: TC-NNN конкретные кейсы ## § Implementation Notes ← Разработчик: trade-offs, deps, file map

Правки (Amendments)

При изменении существующего атома оркестратор классифицирует изменение:

compatible — уточнение формулировки без смены поведения
additive — новый сценарий, новое DR, расширение
breaking — изменение DR, удаление сценария, смена инварианта

Breaking changes требуют impact-анализа: найти все зависимые атомы (parent, consumes, emits), показать пользователю, получить подтверждение, обновить все затронутые атомы, пройти gates заново.

Установка SKILL.md

Чтобы агент заработал по этой методологии, поместите скилл в директорию /mnt/skills/:

структура скилла
atomic-spec-orchestrator/ ├── SKILL.md ← главные инструкции оркестратора ├── references/ │ ├── analyst.md ← детальный reference для Аналитика │ ├── developer.md ← детальный reference для Разработчика │ └── tester.md ← детальный reference для Тестировщика └── assets/ └── atom-template.spec.md ← шаблон нового атома

Агент читает SKILL.md при триггере, затем подгружает нужный references/*.md при переключении роли. Шаблон из assets/ используется при создании нового атома.
