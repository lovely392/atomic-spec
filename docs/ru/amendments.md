# Поправки (Amendments)

Поправки — механизм внесения изменений в атомы, которые уже прошли один или несколько гейтов. Поправки обеспечивают трассируемость изменений и защищают контрактные границы между ролями.

## Три сценария поправок

### Сценарий 1: Атом в статусе Draft (до прохождения всех гейтов)

Атом ещё в работе и не достиг статуса `active`. Изменения вносятся напрямую в исходный файл атома.

**Когда:** Аналитик обнаружил ошибку в DR до прохождения Gate A, или Разработчик нашёл противоречие в AC во время реализации.

**Процесс:**
1. Автор правки создаёт ветку `amend/<ID>-<описание>`.
2. Вносит изменения в исходный файл атома.
3. Обновляет поле `updated` в frontmatter.
4. Добавляет запись в Decision Log с обоснованием изменения.
5. Создаёт PR с чеклистом затронутого гейта.

**Важно:** если изменение затрагивает секции предыдущей роли (например, Разработчик правит AC), требуется ревью от владельца этих секций.

### Сценарий 2: Атом возвращён на доработку (pulled back)

Атом не прошёл гейт и возвращён предыдущей роли.

**Когда:** Ревьюер на Gate B обнаружил, что Tech Spec противоречит AC. Или на Gate C тестировщик нашёл недостающий DR.

**Процесс:**
1. В PR-ревью указываются конкретные пункты чеклиста, которые не пройдены.
2. Автор правит атом в той же ветке.
3. Если нужны изменения в секциях другой роли — создаётся amendment-файл.
4. PR повторно проходит ревью.

### Сценарий 3: Атом зафиксирован в спринте (sprint-locked)

Атом в статусе `active`, его реализация и тесты завершены. Требуется изменение.

**Когда:** Бизнес-требования изменились, обнаружен дефект в production, или нужно расширить функциональность.

**Процесс:**
1. Создаётся **amendment-файл** (отдельный файл поправки).
2. Amendment-файл проходит полный пайплайн (Gate A -> Gate B -> Gate C).
3. После прохождения всех гейтов — изменения вносятся в исходный атом.
4. Если scope изменения значительный — создаётся новый атом с `supersedes`.

## Формат файла поправки

Amendment — это отдельный Markdown-файл со своим YAML frontmatter.

```yaml
---
id: SPEC-042a
type: amendment
title: "Расширение окна отмены заказа до 60 минут"
target: SPEC-042
change-type: ParameterChange
reason: "Аналитика показала, что 78% отмен происходят в интервале 30-45 минут"
status: draft
implementation: none
verification: none
owners:
  analyst: "ivanov"
  developer: "petrov"
  tester: "sidorova"
created: 2025-12-10
updated: 2025-12-10
---

## Что меняется

### Domain Rules

- **DR-002** (изменение): Окно отмены — ~~30 минут~~ **60 минут** с момента `order.placed`.

### Acceptance Criteria

- **AC-002** (изменение): Given заказ создан ~~40~~ **70** минут назад

### Constraints

Без изменений.

## Обоснование

По данным аналитики за Q3 2025, 78% запросов на отмену приходятся на интервал
30-45 минут после оформления. Текущее ограничение в 30 минут приводит к
увеличению нагрузки на поддержку (ручная отмена).

## Impact Analysis

- **Затронутые атомы**: SPEC-042 (целевой), SPEC-043 (дочерний — нотификации)
- **Breaking**: Нет (расширение окна, не сужение)
- **Миграции**: Не требуются (параметр конфигурации)

## Decision Log

- **2025-12-10**: Решение расширить окно с 30 до 60 минут.
  Инициатор: Product Owner. Основание: данные аналитики Q3 2025.
```

## Соглашение об именовании

Amendment-файлы именуются добавлением суффикса к ID целевого атома:

```
SPEC-042.md       — исходный атом
SPEC-042a.md      — первая поправка
SPEC-042b.md      — вторая поправка
SPEC-042c.md      — третья поправка
```

Суффикс — строчная латинская буква, добавляемая последовательно.

**Расположение файлов:**

```
specs/
  orders/
    SPEC-042.md
    SPEC-042a.md
    SPEC-042b.md
```

Amendment-файлы располагаются рядом с целевым атомом в том же каталоге.

## Ветки и коммиты для поправок

```
Ветка:   amend/SPEC-042a-extend-window
Коммит:  amend(SPEC-042a): расширить окно отмены до 60 минут
PR:      [Amend] SPEC-042a: Расширение окна отмены
```

## Принципы работы с поправками

### 1. Атомы не удаляются

Атом никогда не удаляется из репозитория. Вместо удаления используется `status: deprecated`. Новый атом указывает `supersedes: SPEC-042` в frontmatter.

```yaml
# Старый атом SPEC-042
status: deprecated

# Новый атом SPEC-042-v2
supersedes: SPEC-042
status: active
```

### 2. Листья добавляются, ветви меняются минимально

При изменении иерархии атомов предпочитайте добавление дочерних атомов (листьев), а не модификацию существующих (ветвей). Это снижает риск каскадных изменений.

```
# Предпочтительно: добавить лист
SPEC-042
  └── SPEC-042-notify (новый дочерний атом)

# Избегать: переписывать ветвь
SPEC-042 (полностью переработан)
```

### 3. Scope не расширяется

Поправка не может расширять scope исходного атома. Если поправка приводит к тому, что атом начинает описывать новый сценарий — это сигнал к декомпозиции.

Одна поправка — одно конкретное изменение.

### 4. Domain Rules: только добавление (append-only)

Доменные правила не удаляются и не модифицируются напрямую. Вместо этого:

- **Добавление нового DR**: просто добавьте новое правило.
- **Изменение существующего DR**: создайте amendment с явным указанием, что именно меняется.
- **Удаление DR**: пометьте правило как `[DEPRECATED]` и укажите причину в Decision Log.

```markdown
- **DR-002** [DEPRECATED]: Окно отмены — 30 минут.
  Заменено на DR-005 (см. SPEC-042a).
- **DR-005**: Окно отмены — 60 минут с момента `order.placed`.
```

### 5. Open Questions блокируют реализацию, не документацию

Открытый вопрос (`status: open`) блокирует прохождение Gate A и начало реализации, но не блокирует работу Аналитика над документацией. Аналитик может продолжать заполнять секции, фиксируя допущения.

```yaml
open-questions:
  - id: OQ-003
    question: "Нужна ли двойная авторизация для отмены крупных заказов?"
    status: open
    resolution: null
```

Пока OQ-003 открыт, Gate A не может быть пройден. Но Аналитик может написать AC, предполагая ответ, и пометить это в Decision Log.

### 6. Никаких доменных изменений без явного решения

Каждое изменение доменного правила, сценария или ограничения должно быть зафиксировано в Decision Log с указанием:

- Кто принял решение
- На основании чего (данные, требование регулятора, инцидент)
- Какие альтернативы рассматривались

```markdown
## Decision Log

- **2025-12-10**: Расширить окно отмены с 30 до 60 минут.
  Решение: Product Owner (Иванов И.И.)
  Основание: аналитика Q3 2025, 78% отмен в интервале 30-45 мин.
  Альтернативы: 45 минут (отвергнуто — недостаточный запас),
  без ограничений (отвергнуто — логистические риски).
```
