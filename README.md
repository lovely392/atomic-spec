## 🪄 What atomic-spec does

Atomic Spec helps you keep requirements clear, small, and easy to follow. It uses a method that links ideas, use cases, and tests in one place. This helps you keep each part of the work focused on one thing at a time.

The goal is simple:

- Keep one idea in one file
- Keep requirements easy to read
- Keep tests tied to real user needs
- Keep changes easy to track
- Keep teams aligned on what to build

Atomic Spec fits work that uses DDD, TDD, and use-case-driven design. It works well when you need living requirements that stay useful as the product changes.

## 🧭 How to use atomic-spec

Atomic Spec is built around small units of knowledge. Each unit should stay focused on one thing.

A simple flow looks like this:

1. Write the main goal.
2. Break the goal into a use case.
3. Split the use case into small atomic items.
4. Add acceptance points.
5. Link tests to each item.
6. Review each item before you move on.

A good atomic spec file should answer these questions:

- What is the user trying to do?
- What must the system do?
- What should happen if the step fails?
- How do we know the result is correct?

Keep each file short. If a file starts to cover too much, split it into smaller parts.

## 🧱 Core ideas

### One atom

One atom is one small unit of knowledge. It should stay narrow and easy to review. This helps you avoid large blocks of text that are hard to update.

### One file

One file should cover one idea. This makes it easier to find things later and helps avoid confusion when requirements change.

### One unit of knowledge

Each unit should describe one clear piece of behavior. That can be a rule, a case, a test point, or a user need.

### DDD support

Atomic Spec fits domain-driven design because it keeps domain language clear. It helps teams use the same words for the same things.

### TDD support

Atomic Spec also fits test-driven development. You can write the behavior first, then add tests that match it.

### Use-case focus

Use cases help you stay close to real user actions. This keeps the work grounded in what people need, not in vague ideas.

## 🧪 Example workflow

Here is a simple example of how you might use the method:

- User wants to reset a password
- Write the use case for password reset
- Split it into atoms:
  - User enters email
  - System checks the email
  - System sends reset link
  - User opens link
  - User sets new password
- Add checks for each step
- Link each step to a test
- Review the spec when the flow changes

This makes the whole path easier to read and update.

## ✅ Good writing habits

Use these habits when you work with atomic specs:

- Use plain words
- Keep each item short
- Write one fact at a time
- Avoid mixed ideas in one file
- Use the same terms across files
- Tie each requirement to a real action
- Update the spec when behavior changes
